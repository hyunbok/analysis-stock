# 디자인-PRD 비교 분석 스킬

## 목적
app-designer 에이전트가 생성/수정한 화면을 분석하여, `docs/prd.md`에 정의되지 않은 기능이나 요구사항을 `docs/external_prd.md`에 기록한다.

## 실행 절차

### Step 1: 문서 로드
다음 3개 파일을 읽는다:
- `docs/design-concept.md` — 디자인 컨셉 (화면별 레이아웃, UX 패턴, 컴포넌트)
- `docs/prd.md` — 제품 요구사항 (API, DB, 기술 스택, 화면 목록)
- `docs/external_prd.md` — 기존에 기록된 미반영 항목 (있을 경우)

### Step 2: 차이 분석
디자인 컨셉에서 다음 항목들을 추출하여 PRD와 비교한다:

**비교 카테고리:**
1. **API 엔드포인트**: 디자인에서 필요하지만 PRD에 정의되지 않은 API
2. **DB 스키마**: 디자인 화면에 표시되는 데이터 중 PRD DB 스키마에 없는 컬럼/테이블
3. **화면/기능**: 디자인에 포함된 UI 요소 중 PRD 화면 목록이나 기능 설명에 없는 것
4. **사용자 설정**: 디자인의 설정 화면에 있지만 PRD에 정의되지 않은 설정 항목
5. **인터랙션 패턴**: 디자인에서 정의한 UX 패턴 중 백엔드 지원이 필요하지만 PRD에 없는 것

**비교 시 주의사항:**
- PRD에 "상세는 구현 태스크에서 정의" 같은 위임 표현이 있는 항목은 미반영으로 간주하지 않는다
- 순수 클라이언트 측 UI 패턴 (애니메이션, 제스처 등)은 PRD 비교 대상에서 제외한다
- 이미 `external_prd.md`에 기록된 항목은 중복 기록하지 않는다

### Step 3: external_prd.md 업데이트
새로 발견된 항목이 있으면 `docs/external_prd.md`에 추가한다.

**각 항목의 기록 형식:**
```markdown
### N.N 항목명
- **디자인 컨셉**: 해당 내용이 있는 섹션과 설명
- **PRD 현황**: PRD에서 관련 내용이 있는지, 없는지
- **필요 작업**: PRD에 반영 시 필요한 구체적 작업 (API, DB, 서비스 등)
- **우선순위 제안**: Phase 1 또는 Phase 2
```

**요약 매트릭스도 업데이트한다:**
- 새 항목을 Phase 1/Phase 2 매트릭스 테이블에 추가
- 합계 행 갱신

### Step 4: 결과 보고
사용자에게 다음을 보고한다:
- 새로 발견된 미반영 항목 수
- 각 항목의 간략한 요약 (1줄)
- Phase 1/Phase 2 분류

## 참고
- 이 스킬은 app-designer 에이전트 완료 후 자동으로 호출되도록 hook이 설정되어 있다
- 수동으로도 `/compare-design-prd` 명령으로 실행할 수 있다
